# 极简健身记录 App 开发日志

本文档用于记录项目从零到 MVP 的每一步开发过程，包括需求决策、目录结构、关键实现、验证结果和下一步计划。后续每完成一个阶段，都优先更新本文档。

如果需要更直观地查看当前状态，请看：[MVP 进度地图](./progress-map.md)。

## 项目目标

构建一款主打极简主义、零摩擦打卡的移动端健身记录 App。

核心体验原则：

- 大面积留白，去除多余分割线、阴影和花哨渐变。
- 使用高品质、轻量、克制的排版风格。
- 避免默认键盘输入，优先使用大号控件完成交互。
- 所有关键转场、缩放、滚动反馈使用 `react-native-reanimated` 的弹性动画。
- App 需要支持中文和英文界面切换。

## 技术栈约定

前端：

- React Native
- Expo
- Expo Router
- react-native-reanimated
- 简洁 StyleSheet，暂不引入臃肿 UI 组件库
- 自建轻量 i18n，不引入额外翻译框架

后端：

- FastAPI
- SQLAlchemy
- SQLite 作为 MVP 本地开发数据库
- Pydantic schema

## 训练记录导入 MVP

完成时间：2026-04-28

背景：

- 现有用户可能已经在训记、表格或备忘录里积累了大量训练记录，如果只能从零开始，会显著提高切换成本。
- 当前产品里已经有日历档案和按日详情，因此最适合优先落地的是“训练记录导入”，而不是先导入体重、心率一类暂时没有展示承接面的数据。

操作内容：

- 在设置页新增 `导入训练记录` 入口，并保留一个很轻的二级流程：`文件导入` 与 `截图导入`。
- 保留截图导入为后续入口，但先把 `文件导入` 做成可试用 MVP，支持从文件选择器读取 `CSV / TSV / TXT`，也支持直接粘贴文本记录。
- 新增训练记录解析模块 `src/import/workoutImport.ts`，用于识别常见表头别名（如日期 / 动作 / 组数 / 重量 / 次数 / 完成情况），并在缺少表头时按常见列顺序兜底。
- 为动作名称增加肌群推断逻辑：优先匹配现有动作字典；若没有精确命中，则根据常见关键词（如卧推、划船、深蹲、弯举、卷腹等）推断到 `胸 / 背 / 臀腿 / 手臂 / 腹部 / 肩`。
- 在训练上下文里新增 `importDraft`、`setImportDraft`、`clearImportDraft`、`applyImportDraft` 与 `revealArchiveDate`，让导入流程可以先暂存草稿，再在确认时一次性写入本地训练记录。
- `导入预览` 页面不再展示假数据，而是读取真实草稿，动态显示：
  - 总记录数
  - 动作数量
  - 时间范围
  - 前几条训练样例
- 用户点击 `确认导入` 后，会把训练记录写入本地 session 存储，并回到首页自动展开最新导入日期，方便立即检查导入结果是否正确。
- 同步收紧导入页文案：把原来“先确认页面与流程，再接真实导入”之类的原型文字改成“文件导入已可试用，截图导入稍后接入”，避免界面状态和真实能力不一致。
- 同轮还按最新要求微调设置页头部：删除顶部 `training day` 小字，并把 `设置` 标题字重提升到更稳的粗度。

当前边界：

- 当前 MVP 先支持结构较清晰的文本型训练记录，不包含 Excel 二进制解析。
- `截图导入` 仍是占位入口，尚未接入 OCR 或图片识别。
- 导入逻辑默认把记录视为“已完成训练”，并写入首页日历与当天详情；更复杂的计划态、次数态与多字段校验可在下一阶段继续补强。
- 首页日历浏览范围新增下边界：最早只能切换回 `2024 年 1 月`，不会再无限向过去翻，先和当前种子数据与产品阶段保持一致。
- 文件导入页移除了原型阶段的“字段映射”可视区块，改成更轻的示例说明，让用户优先用“选文件 / 粘贴文本 -> 预览 -> 导入”的主路径试用，不再像填写报表。
- 同时放宽了训练文本解析规则：除了 `CSV / TSV` 这类结构化内容外，也支持更自然的纯文本行，例如 `2026-04-21 卧推 4组 60kg`。系统会优先识别日期和动作，再从尾部提取组数与重量，降低用户必须“按模板一比一输入”的门槛。
- 文件导入新增 `.xlsx` / `.xls` 支持：通过读取工作簿首个 sheet 并转成文本表格后复用现有解析链路，让用户可以直接从 Excel 导入，而不必先手动另存成 CSV。
- 继续增强“自由文本”解析：新增对 `4组60公斤卧推`、`卧推 60kg 4组`、`4x60kg 卧推` 这类更接近日常记录习惯的顺序识别。系统会先拆出日期，再识别 `组数 × 重量`、显式单位和剩余动作名，进一步降低格式束缚感。
- 为“导入预览”补上手动修正层：用户现在可以直接点按某一条预览记录，弹出一个轻量编辑面板，逐条修正 `日期 / 动作名 / 组数 / 重量 / 肌群` 后再确认导入。这样即使 OCR、Excel 表头或文本推断出现偏差，也不需要重新准备整份文件，只改出错的那几条即可。
- 将“截图导入”从纯占位页升级为可试用 MVP：现在可以从相册选择训练截图，在页面内触发 OCR 识别，把识别出的文字先落进可编辑文本框，再沿用现有的“导入预览 -> 手动修正 -> 确认导入”链路继续完成迁移。
- 为了兼顾可用性与稳定性，截图导入采用“两段式”流程：`选图 -> OCR -> 文本可编辑 -> 预览修正 -> 导入`。这样即使识别结果不完美，用户也只需要改文字和个别记录，而不用整张截图重来。
- 当前截图识别基于 WebView 中运行的浏览器 OCR 脚本，因此首次识别会比普通文本导入更慢一些，也更依赖网络把 OCR 模型拉下来；但作为 Expo Go 阶段的可试用方案，它已经足够让用户在手机上验证这条迁移路径的手感。
- 继续优化截图识别结果的可用性：OCR 返回后不再把原始文本整段扔给用户，而是先经过一层“有效训练记录筛选”。系统会优先保留能解析出 `日期 + 动作 + 组数/重量` 的内容，并自动整理成统一的训练记录文本；如果一条有效记录都没提取出来，就直接提示“没有识别到有效训练记录”。
- 在用户进一步体验后，决定先把“导入训练记录”从设置页中隐藏，不再作为当前 MVP 的对外能力暴露。原因不是底层思路彻底放弃，而是这条线在现阶段仍然偏重、偏复杂，会拉低整款 App 当前极简且高完成度的气质。现阶段只隐藏入口，底层代码保留，后续若要重启这条能力，可以在不从零搭建的前提下继续打磨。
- 为“目标”页补上 `自定义` 分支，并删除页内的“第一步”提示：用户现在除了胸 / 肩 / 背 / 臀腿 / 手臂 / 腹部外，还可以直接选择 `自定义`，在下一页输入今天要做的动作名称。自定义动作会被编码成与普通动作一致的内部结构，因此仍然能沿用现有的 `今日动作 -> 设置 -> 执行` 主链，而不是另起一套临时流程。这样既解决了“动作库永远不可能覆盖所有训练习惯”的问题，也保持了训练路径的一致性。

验证结果：

- `npm run typecheck` 通过。

## 训练完成后的导航栈清理

完成时间：2026-04-23

背景：

- 用户在真机测试时发现，完成一个动作后点击返回，可能回到训练前的组数/重量配置页。
- 训练一旦开始，配置页就应被视为已经消费掉的临时步骤，不应保留在后退路径中。

操作内容：

- 将配置页点击 `START` 进入执行页的导航从 `router.push` 改为 `router.replace`。
- 单动作训练完成后使用 `router.dismissTo("/")` 回到首页，清理训练流程栈。
- 多动作训练全部完成后，“完成训练”同样使用 `router.dismissTo("/")` 回到首页。
- 多动作单个动作完成后仍返回动作队列，但配置页不会出现在返回栈中。
- 多动作队列进入配置页改为 `router.replace`，避免旧的 `0/3` 队列状态残留在返回栈中。
- 多动作配置页返回按钮主动 `replace` 回当前队列，并关闭配置页边缘返回手势，避免绕过队列状态同步。
- 今日动作全部完成后，队列页进入终态：隐藏返回按钮、关闭边缘返回手势、列表卡片不可点，只保留“完成训练”作为唯一操作。
